# 우리 팀 Task 관리 전략

> Issue · Branch · PR · Commit 운영 규칙. 모든 협업은 이 문서를 기준으로 한다.

---

## 1️⃣ Issue 관리 전략

Issue는 **명확한 완료 기준(Definition of Done)을 정의할 수 있는 작업**을 기준으로 생성한다.

- 작업 진행 상황을 쉽게 추적할 수 있도록 최대한 **세분화**하여 관리한다.
- 하나의 Issue는 **하나의 목표**를 가지도록 작성한다.

### ❓ 하위 Issue가 너무 많아지는 경우

- 관리 비용이 과도하게 증가할 경우, **관련성이 높은 작업들을 묶어서** 관리한다.
- 주제 또는 기능 단위로 **2~4개의 Issue를 하나의 상위 작업**으로 구성할 수 있다.

### 계층 구조

```text
상위 Issue (Epic / 모듈·기능)      ← 추적용 컨테이너, Branch 없음
  └ 그룹 Issue (관련 작업 묶음)     ← 2~4개 묶을 때 (선택)
      └ 작업 Issue                ← 실제 작업 단위 = 1 PR ★
          └ 체크리스트 (- [ ])     ← Issue 본문, commit 단위
```

---

## 2️⃣ Branch 생성 기준

Issue의 규모가 충분히 크다면 **Issue별로 Branch를 생성하여 작업**한다.

- 작업 완료 후 Pull Request를 통해 **`dev` 브랜치에 병합**한다.
- 병합이 완료된 Branch는 **삭제**한다.

Issue의 규모가 작다면, 관련된 **2~4개의 Issue를 하나의 Branch로 묶어서** 작업할 수 있다.

### 브랜치 네이밍

```text
<type>/<이슈번호>-<짧은-설명>

feat/6-signup
fix/27-token-expiry
docs/5-readme
```

- 분기 대상: **`dev`** (통합 브랜치)
- `main`은 안정/배포 라인 — 직접 작업하지 않는다.

---

## 3️⃣ PR (Pull Request) 기준

우리 팀은 **1 Branch = 1 PR** 원칙을 사용한다.

- 하나의 Branch는 **하나의 작업 목적**만 가지도록 유지한다.
- PR에는 해당 Branch에서 수행한 작업만 포함한다.
- 리뷰 및 이력 추적이 용이하도록 **불필요하게 많은 작업을 하나의 PR에 포함하지 않는다.**
- PR 본문에 `Closes #이슈번호` 를 적어 머지 시 Issue가 자동으로 닫히게 한다. (여러 개면 `Closes #6, Closes #7`)
- 병합 대상은 **`dev`**.
- 병합 방식은 **Merge commit 또는 Rebase**를 사용한다. **Squash 금지** — 커밋 단위 이력(이슈 ↔ 커밋 연결)을 보존하기 위함.

### 한 PR 크기 테스트 — 묶기/나누기 기준

> **이 묶음이 한 PR(~400줄, 1~3일, 응집된 변경)에 들어가는가?**
>
> - **YES** → 관련 Issue 2~4개를 1 Branch = 1 PR로 묶는다.
> - **NO** → 한 그룹이라도 2개 이상의 PR로 나눈다.

- 그룹(상위로 묶인 Issue)은 **추적 단위**일 뿐, 자동으로 PR 경계가 아니다.
- 작고 응집적인 묶음(예: 회원가입+로그인)은 1 PR로 묶고, 각 항목이 독립적으로 큰 묶음(예: 게시글/댓글/태그/검색)은 나눈다.
- 큰 PR은 리뷰가 형식적(고무도장)이 되므로 **작게 유지**한다.

## 4️⃣ PR 컨벤션

### PR 제목 형식

```text
<type>(<scope1>,<scope2>): <변경 내용 요약>
```

- `type`은 Commit 컨벤션의 타입과 동일하게 사용한다.
- PR 제목에는 대표 `type` 하나만 작성한다.
- `scope`는 기능/도메인과 변경 영역을 혼합해서 작성한다.
- `scope`는 작업한 모든 범위를 반영하되, 너무 길어지지 않도록 실제 리뷰 범위를 설명하는 핵심 scope만 작성한다.
- PR 제목과 본문은 한국어로 작성한다. 단, `type`, `scope`, API 이름, 에러 메시지, 외부 서비스명처럼 원문 유지가 필요한 표현은 그대로 사용한다.
- Issue 번호는 제목에 작성하지 않고, PR 본문에 작성한다.
- 환경 변수, DB 변경, 마이그레이션 등 배포 영향이 있는 경우 PR 제목 맨 앞에 `🚨`를 붙인다.
- `🚨`가 붙은 PR은 작성자가 영향 받는 도메인 담당자에게 PR 생성 전 또는 생성 직후 공유한다.
- 공통 문서, 작업 규칙, API 계약, DB schema, 배포 설정처럼 팀 전체에 영향을 주는 변경은 관련 담당자를 PR 본문 또는 코멘트에서 명시적으로 언급한다.

예시:

```text
feat(auth,api,db): 회원가입 기능 추가
fix(token,api): access token 만료 처리 수정
docs(readme): 실행 방법 문서 수정
🚨 feat(auth,api,db): 회원가입 DB 마이그레이션 추가
```

### PR 본문 템플릿

```md
## 작업 내용

## 관련 Issue

Closes #

## 주요 변경 사항

## 테스트

- `테스트 명령어 또는 확인 내용` → 결과
- `미수행한 테스트` → 미수행: 사유

## 리뷰 포인트

## 체크리스트

- [ ] PR 제목이 컨벤션에 맞게 작성되었다.
- [ ] 관련 Issue가 본문에 `Closes #이슈번호` 형식으로 연결되었다.
- [ ] PR 범위가 하나의 작업 목적에 맞게 유지되었다.
- [ ] 불필요한 변경사항이 포함되지 않았다.
- [ ] 공통 문서 또는 자신의 작업 범위를 벗어난 변경이 포함되지 않았는지 확인했다.
- [ ] 로컬에서 빌드/실행을 확인했다.
- [ ] 필요한 테스트를 수행했고 결과를 작성했다.
- [ ] 리뷰어가 중점적으로 봐야 할 내용을 작성했다.
- [ ] 문서 수정이 필요한 경우 반영했다.
- [ ] UI 변경이 있는 경우 스크린샷 또는 동작 설명을 첨부했다.
- [ ] 환경 변수, DB 변경, 마이그레이션 등 배포 영향이 있는 경우 명시했다.
- [ ] 배포 영향이 있는 경우 PR 제목 맨 앞에 `🚨`를 추가했다.
- [ ] `🚨` PR인 경우 영향 받는 도메인/Infra/DB 담당자에게 사전 또는 PR 생성 직후 공유했다.
```

### 선택 섹션

필요한 경우에만 다음 섹션을 추가한다.

```md
## 스크린샷 / 동작 설명

## 배포 영향
```

선택 섹션은 모든 PR에 강제하지 않는다. UI 변경이나 배포 영향이 있는 경우에만 작성한다.

### 테스트 작성 기준

`테스트` 섹션에는 성공, 실패, 미수행 여부와 사유를 함께 작성한다.

예시:

```md
## 테스트

- `npm test` → 통과
- 회원가입 플로우 수동 확인 → 정상
- E2E 테스트 → 미수행: 테스트 환경 미구성
```

테스트를 수행하지 못한 경우에도 항목을 생략하지 않고 미수행 사유를 남긴다.

### 리뷰 요청 기준

PR은 작성자 기준으로 작업이 완료된 뒤 생성한다.

리뷰 요청 전에는 다음을 완료해야 한다.

- 빌드/실행 확인
- 필요한 테스트 수행 및 결과 작성
- PR 체크리스트 작성
- 작성자 self review 완료

리뷰어가 중점적으로 봐야 할 내용이 있다면 `리뷰 포인트`에 작성한다.

### Draft PR 기준

- Draft PR은 사용하지 않는다.
- PR은 작성자 기준으로 작업, 확인, self review가 완료된 뒤 생성한다.

### 머지 조건

- 작성자가 PR 체크리스트를 모두 완료하면 `dev`에 머지할 수 있다.
- 리뷰어 승인은 필수 조건으로 두지 않는다.

## 5️⃣ Commit 컨벤션

### 타입

| 타입       | 내용                                                  |
| ---------- | ----------------------------------------------------- |
| `feat`     | 새로운 기능에 대한 커밋                               |
| `fix`      | 버그 수정에 대한 커밋                                 |
| `build`    | 빌드 관련 파일 수정 / 모듈 설치 또는 삭제에 대한 커밋 |
| `chore`    | 그 외 자잘한 수정에 대한 커밋                         |
| `ci`       | CI 관련 설정 수정에 대한 커밋                         |
| `docs`     | 문서 수정에 대한 커밋                                 |
| `style`    | 코드 스타일 혹은 포맷 등에 관한 커밋                  |
| `refactor` | 코드 리팩토링에 대한 커밋                             |
| `test`     | 테스트 코드 수정에 대한 커밋                          |
| `perf`     | 성능 개선에 대한 커밋                                 |

### 커밋 메시지 형식

```text
<type>: <변경 내용 요약> (#이슈번호)

예) feat: JWT access 토큰 발급 로직 추가 (#7)
```

- 커밋 메시지의 변경 내용 요약은 한국어로 작성한다. 단, `type`은 위 표의 영어 키워드를 사용하고, API 이름이나 기술 용어처럼 원문 유지가 필요한 표현은 그대로 사용한다.

### 커밋 기준

- **이슈의 할 일 목록(체크리스트) 하나마다 commit 생성.**
- 체크리스트 작업 하나가 끝날 때마다 해당 내용에 대한 commit을 작성한다.
- 커밋 메시지 끝에 **이슈 번호 `(#N)`** 를 붙여 커밋 ↔ 이슈를 자동 연결한다. (GitHub 이슈 타임라인에 커밋이 표시되어, PR에 매핑 표를 따로 쓰지 않아도 추적 가능)
